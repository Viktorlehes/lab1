
    1.  Bör ni använda implementationsarv (subclassing) eller specifikationsarv (interfaces)?
        Varför? Kan båda användas? Vilka för- och nackdelar ser ni?

    Både en parent klass och/eller en interface kan användas för att implementera Car klassen.
    Fördelen med att använda en parent class är att det finns många delade fields och även någon metod som
    är likadan i båda klasser som kan abstraheras till parent klassen. Nackdelen med detta är att det finns
    t.ex metoden speedFactor som delas i båda klasserna och bör override:as men blir otydligt att man ska
    implementera själv om den definieras i Car.
    Därför är ett interface också lämpligt att använda då vi kan kräva metoden speedFactor utan implementation
    samt implementera default funktioner som kan användas i alla subclasser. Nackdelen är att vi inte kan
    implementera något annat än konstanter fields och måste definiera våra fields duplicerat i sub class.

    Därför bör abstract class användas. Vi kan definiera våra fields samt getter / setter funktioner som
    erhåller sig till open/close principle. Vi kan också implementera våra delade metoder men även
    abstract metoder som kräver implementation i subclass.

    2.  Vilken synlighet bör ni använda för de olika metoder och variabler som klasserna innehåller?
        Vilket gränssnitt bör ni exponera?

    Alla fields skall vara private enligt open/closed principle och för att undvika aliasing problem.
    Getter funktioner kan vara public, men interna setter funktioner kan vara protected sådan att de endast
    inom subclasses eller package nivå.
